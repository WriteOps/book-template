# Project Structure and Conventions

To keep your book organized and ensure the build script works correctly, it's important to follow the project's folder and file naming conventions.

## The Chapter System

The build script automatically finds all your content by looking inside the `chapters/` folder.

### Folder Naming
Each chapter must be in its own folder named `chapter-{n}`:
- `chapters/chapter-1/`
- `chapters/chapter-2/`
- `chapters/chapter-10/`

**Note:** The script uses "natural sorting," so `chapter-10` correctly comes after `chapter-9`.

### File Naming
Inside each folder, you can have multiple Markdown files. The script will concatenate them in alphanumeric order. We recommend using an index:
- `1.introduction.md`
- `2.deep-dive.md`
- `3.summary.md`

## The Meta Directory

The `meta/` directory contains files that aren't part of the main numbered chapters but are essential for a professional book:

1.  **`title-copyright.tex`**: A raw LaTeX file for the title and copyright pages. This renders *before* the Table of Contents.
2.  **`preface.md`**: The introduction to your book.
3.  **`conclusion.md`**: The final chapter.
4.  **`about-author.md`**: Your biography and photo.

## Assets and Images

Keep all your media in the `assets/` directory. By convention, images go in `assets/images/`.

When linking images in your chapters, use paths relative to the file so they work in your editor's preview:
`![My Image](../../assets/images/my-image.png)`

### Remote Assets
The build pipeline supports remote URLs (e.g. `https://...`) for images anywhere in your book (chapters or metadata). They are automatically downloaded and cached locally during the build process to ensure LaTeX compatibility.

```markdown
![My Image](https://raw.githubusercontent.com/WriteOps/.github/refs/heads/main/logo.png)
```
